## 학습 내용 기록

### 1. GEO란 무엇인가

SEO(Search Engine Optimization)가 구글 검색 결과 상위 노출을 목표로 한다면,  
GEO는 ChatGPT, Perplexity, Claude 같은 LLM이 답변을 생성할 때 내 콘텐츠를 인용하도록 최적화하는 전략이다.

```
SEO: 구글 크롤러가 내 페이지를 높게 평가하도록 최적화
GEO: LLM이 내 콘텐츠를 인용하도록 최적화
```

LLM이 콘텐츠를 인용하는 데 영향을 주는 요소:
- 가독성 (Readability)
- 전문성 (Expertise)
- 신뢰성 (Trustworthiness)
- FAQ 구조 여부
- 요약 가능성

---

### 2. GEO 스코어링

LLM 평가 점수(70점)와 HTML 메트릭 점수(30점)를 합산해 최종 GEO 점수를 산출한다.

**LLM 점수 (70점)**

| 항목 | 가중치 |
|------|--------|
| 전문성 (expertise) | 30% |
| 가독성 (readability) | 25% |
| 신뢰성 (trustworthiness) | 25% |
| FAQ 구조 | 10% |
| 요약 가능성 | 10% |

LLM에게 콘텐츠를 넘기고 각 항목을 0~10점으로 평가받은 뒤 가중합산.

**HTML 메트릭 점수 (30점)**

| 항목 | 점수 | 기준 |
|------|------|------|
| Title 길이 | 5점 | 30~60자 |
| Meta Description | 5점 | 120~160자 |
| H1 태그 | 5점 | 정확히 1개 |
| 헤딩 구조 비율 | 3점 | heading_ratio >= 0.2 |
| 단락 평균 길이 | 3점 | 30~120자 |
| 이미지 ALT 비율 | 4점 | alt 있는 이미지 비율 |
| 단어 수 | 5점 | 800단어 이상 |

### 3. 검색엔진 기초

LLM이 특정 콘텐츠를 인용하는 원리를 이해하기 위해 검색엔진 내부 구조를 학습했다.

#### 역색인 (Inverted Index)

"문서 → 단어 목록" 대신 "단어 → 문서 목록"으로 저장해 검색 속도를 높이는 자료구조.

```python
# 일반 색인 (Forward Index)
{문서1: ["GEO", "검색", "최적화"], 문서2: ["파이썬", "검색"]}

# 역색인 (Inverted Index)
{"GEO": {문서1}, "검색": {문서1, 문서2}, "최적화": {문서1}}
```

역색인이 있으면 "검색"이라는 단어가 포함된 문서를 O(1)에 가까운 속도로 찾을 수 있다.  
단, "어떤 문서가 더 관련도 높은가"는 알 수 없어서 TF-IDF/BM25가 필요하다.

#### TF-IDF

관련도 점수를 계산하는 알고리즘. 두 값의 곱으로 계산한다.

```
TF (Term Frequency)
= 이 문서 안에서 이 단어가 얼마나 자주 나오는가
= 단어 등장 횟수 / 문서 전체 단어 수

IDF (Inverse Document Frequency)
= 전체 문서 중 이 단어가 얼마나 드문가
= log(전체 문서 수 / 이 단어가 등장한 문서 수)

TF-IDF = TF × IDF
```

자주 나오면서(TF) 동시에 희귀한(IDF) 단어일수록 점수가 높아진다.  
"은/는/이/가" 같은 흔한 단어는 IDF가 낮아서 자동으로 영향이 줄어든다.

**한계**: 문서 길이 보정이 부족하다.  
짧은 문서는 TF가 부풀려져서 실제보다 관련도가 높게 나온다.

#### BM25

TF-IDF의 실전 개선판. Elasticsearch 기본 알고리즘.

```
BM25 = Σ IDF(w) × TF(w,d)×(k1+1) / (TF(w,d) + k1×(1 - b + b×|d|/avgdl))

k1 = 1.5  (TF 포화 속도, 높을수록 반복 등장의 영향이 오래 유지)
b  = 0.75 (문서 길이 보정 강도, 1에 가까울수록 강하게 보정)
```

TF-IDF 대비 두 가지 개선:

1. **TF 포화**: 단어가 아무리 많이 나와도 점수가 무한정 올라가지 않음
2. **문서 길이 정규화**: 짧은 문서가 부당하게 유리한 걸 보정

실험 결과 — 쿼리 `"GEO 검색 최적화"`:
```
TF-IDF: 3번(짧은 문서) > 1번(긴 문서)  ← 길이 왜곡
BM25:   1번(긴 문서)   > 3번(짧은 문서) ← 올바른 랭킹
```

### 4. Elasticsearch (개요)

우리가 직접 구현한 역색인 + BM25를 프로덕션 수준으로 자동화해주는 분산 검색 엔진.

| 우리 코드 | ES 자동화 |
|-----------|-----------|
| `tokenize()` | Analyzer (언어별 내장) |
| `build_inverted_index()` | Lucene 내부 자동 처리 |
| `compute_bm25_scores()` | 기본 유사도 엔진 |
| `search_bm25()` | Query DSL |
| `dict[int, list[str]]` | Index + Mapping |

추가로 해주는 것: 디스크 영속성, 분산 처리, 복합 쿼리, 집계(Aggregation), 벡터 검색(kNN).

### 5. 임베딩 (Embedding)

텍스트를 숫자 벡터로 변환해서 의미적 유사도를 계산할 수 있게 만드는 기술.

```
BM25:  "GEO 최적화"와 "LLM 인용 전략" → 키워드 불일치 → 관련 없음
임베딩: "GEO 최적화"와 "LLM 인용 전략" → 벡터 가까움  → 의미 유사
```

**발전 과정**

```
One-Hot Encoding  → 단어 위치만 표시, 의미 없음
Word2Vec          → 문맥으로 의미 학습, 단어 단위 고정 벡터
Transformer       → 문장 전체 문맥 반영, 현재 표준
```

**Transformer 학습 원리**

1. 문장에서 일부 단어를 가리고 맞추는 문제(빈칸 맞추기)를 수억 번 반복
2. Self-Attention으로 문장 내 모든 단어가 서로를 참조
3. 비슷한 문맥에 등장하는 단어 → 비슷한 벡터로 수렴
4. 결과적으로 의미가 가까운 텍스트 = 벡터 공간에서 가까운 위치

```python
# OpenAI 임베딩 API
response = openai_client.embeddings.create(
    model="text-embedding-3-small",
    input="GEO 분석기로 검색 노출을 최적화합니다"
)
vector = response.data[0].embedding  # 1536차원 float 배열
```

**유사도 계산**: 코사인 유사도 (-1 ~ 1, 1에 가까울수록 유사)

```python
def cosine_similarity(a, b):
    return np.dot(a, b) / (np.linalg.norm(a) * np.linalg.norm(b))
```

### 6. 벡터DB

임베딩 벡터를 저장하고, 의미적으로 가장 가까운 벡터를 빠르게 찾아주는 데이터베이스.

**일반 DB vs 벡터DB**
```
일반 DB: WHERE content LIKE '%GEO%'  → 키워드 일치만
벡터DB:  쿼리 벡터와 가장 가까운 벡터 탐색 → 의미 유사도
```

**근사 최근접 탐색 (ANN) 알고리즘**

벡터 수백만 개를 전부 비교하면 느리기 때문에 근사 알고리즘을 사용한다.

- **IVF**: 벡터 공간을 클러스터로 나눠 가까운 클러스터만 탐색. 메모리 효율적.
- **HNSW**: 계층적 그래프로 위에서부터 좁혀 내려가며 탐색. 현재 가장 널리 사용.

이 프로젝트 규모(수백~수천 문서)에서는 Chroma 기본값(HNSW)으로 충분하다.

## 기술 선택 이유 요약

```
역색인/BM25  → 키워드 기반 후보 문서 추출 원리 이해
임베딩       → 의미 기반 유사도 계산
벡터DB       → 임베딩 벡터 저장 및 빠른 탐색

RAG          → LLM이 실제로 콘텐츠를 인용하는 파이프라인 재현
```

GEO 분석기의 핵심 질문은 **"내 콘텐츠가 LLM의 RAG 파이프라인에서 선택될 가능성이 얼마나 되는가"**이다.  
이 파이프라인을 직접 이해하고 구현하기 위해 위 기술들을 학습했다.
